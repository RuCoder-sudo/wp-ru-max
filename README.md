## О плагине

### Основные возможности

#### Автопубликация
- Новая или обновлённая запись / страница → автоматически отправляется в **канал MAX**.
- Поддержка миниатюры (изображение прикрепляется к сообщению).
- Возможность отключить ссылку «Читать полностью».

#### Личные уведомления
- Перехват всех email-уведомлений WordPress.
- Поддержка **WooCommerce** (новые заказы), **Contact Form 7**, **Elementor Forms**, **Gravity Forms**, **Ninja Forms** и любых других форм, использующих `wp_mail()`.
- Каждое письмо дублируется в личный чат с ботом MAX.

#### Чат-виджет
- Плавающая кнопка MAX на сайте.
- Анимация «печатания» для приветственного сообщения.
- Полная кастомизация внешнего вида (через настройки плагина).

#### История и логи
- Полная таблица всех событий (отправки, ошибки, тесты).
- Фильтрация по типу, дате, статусу.
- Просмотр подробного JSON-лога каждого запроса.

**WP Ru-max** подключает WordPress-сайт к мессенджеру [MAX (max.ru)](https://max.ru) и решает три задачи:

- **Автопубликация** — новые записи и страницы автоматически отправляются в канал MAX.
- **Уведомления** — все email WordPress (заказы WooCommerce, заявки с форм) дублируются в личный чат с ботом.
- **Чат-виджет** — настраиваемая плавающая кнопка MAX на сайте для связи с посетителями.

Лицензирование и управление ключами реализовано через REST API сайта [рукодер.рф](https://рукодер.рф).

## Установка

### Способ 1: через панель WordPress

1. Скачайте ZIP-архив плагина
2. Перейдите в **Плагины → Добавить новый → Загрузить плагин**
3. Выберите файл `wp-ru-max.zip` и нажмите **Установить**
4. Нажмите **Активировать плагин**

### Способ 2: через FTP / SSH

1. Клонируйте репозиторий или распакуйте архив:
   ```bash
   git clone https://github.com/RuCoder-sudo/wp-ru-max.git
   ```
2. Загрузите папку `wp-ru-max` в `/wp-content/plugins/`
3. Активируйте плагин в **Плагины → Установленные плагины**

## Настройка

### 1. Активация лицензии

1. Перейдите в **Ru-max → Активация**
2. Введите лицензионный ключ (формат `WPRM-XXXX-XXXX-XXXX-XXXX`)
3. Нажмите **Активировать**

Нет ключа — заполните форму «Запросить лицензионный ключ» на той же вкладке.

### 2. Подключение бота MAX

1. Получите токен бота на платформе [MAX для партнёров](https://max.ru/partner):  
   «Чат-боты» → «Интеграция» → «Получить токен»
2. Вставьте токен на вкладке **Ru-max → Главная**
3. Нажмите **Проверить подключение**

### 3. Автопубликация

На вкладке **Публикация**:
- Включите автоотправку и укажите ID или никнейм канала MAX
- Настройте формат сообщения и включение/отключение ссылки

### 4. Уведомления

На вкладке **Уведомления**:
- Включите перехват `wp_mail()`
- Опционально — укажите фильтр тем для выбора нужных уведомлений

### 5. Чат-виджет

На вкладке **Виджет**:
- Включите виджет и укажите ссылку на бота или канал
- Настройте текст приветствия, анимацию, звук, задержку и отступ

## Как работает автообновление

Плагин проверяет [GitHub Releases](https://github.com/RuCoder-sudo/wp-ru-max/releases) раз в 12 часов. Если тег нового релиза новее установленной версии — в WordPress появляется стандартное уведомление «Доступно обновление». Обновление устанавливается в один клик, как обычный плагин из WordPress.org.

## Changelog

= 1.0.24 =
* Исправлено (критично, версия 1.0.24 — окончательное решение): тумблер «Автоотправка в MAX» в Гутенберге сбрасывался в ВЫКЛ после сохранения, и в логах не появлялось НИ ОДНОЙ записи. Гутенберг по неизвестной причине не присылал meta-поле в REST-запросе. Решение — отказ от стандартной схемы «meta-через-REST» в пользу собственного REST-эндпоинта:
  * Добавлены маршруты `GET/POST /wp-json/wp-ru-max/v1/skip/{post_id}` (плюс AJAX-fallback на случай блокировки REST файрволом).
  * JS-панель читает текущее состояние тумблера ИЗ БД через свой эндпоинт при открытии статьи.
  * При клике по тумблеру состояние сохраняется СРАЗУ, не дожидаясь кнопки «Сохранить». Контрольное чтение из БД пишется в лог.
  * При ошибке UI откатывает тумблер обратно и показывает сообщение.
* Исправлено (критично, версия 1.0.24): тумблер «Автоотправка в MAX» при включении (положение ВКЛ) сбрасывался обратно в ВЫКЛ после сохранения статьи. Корневая причина — `default = '1'` в `register_post_meta` приводил к специальной обработке WordPress при сохранении значения, совпадающего с дефолтом. Что сделано:
  * Удалён `default` из `register_post_meta` (и из `show_in_rest.schema`). Дефолт «ВЫКЛ» реализован на уровне ЧТЕНИЯ.
  * В `persist_skip_meta_from_rest` использован атомарный `update_post_meta`. Сразу после записи делается контрольное чтение из БД (с очисткой кэша) и пишется в лог: «пришло X → нормализовано Y → в БД Z (ВКЛ/ВЫКЛ)».
  * `sanitize_skip_meta` переписан: только `'0'` (включая `0`/`false`) трактуется как «ВКЛ»; всё остальное — `'1'` («ВЫКЛ»).
  * В `gutenberg-panel.js` сравнение мета-значения сделано мягким: `String(raw).trim() === '0'` — корректно отрабатывает строку `'0'`, число `0`, `false`.
  * В `WP_Ru_Max_Post_Sender::on_post_status_change` чтение значения тоже сделано мягким (`is_scalar` + `trim`).
* Исправлено (критично, окончательно): переключатель «Автоотправка в MAX» в редакторе Гутенберг наконец-то стабильно сохраняется в положениях ВКЛ/ВЫКЛ. Корневая причина — мета-ключ `_wp_ru_max_skip` начинался с подчёркивания и трактовался WordPress как «protected meta». REST API (Гутенберг) такие ключи фактически не обновлял, даже при `register_post_meta` + `auth_callback`. Параллельно чтение `php://input` в `save_post` было ненадёжно — REST-сервер уже мог разобрать тело запроса. Сделаны следующие изменения:
  * Мета-ключ переименован на `wp_ru_max_skip` (без подчёркивания) — больше никаких protected-meta.
  * Добавлена одноразовая миграция значений из старого ключа `_wp_ru_max_skip` в `wp_ru_max_skip` (флаг `wp_ru_max_skip_meta_migrated_v1` в `wp_options`).
  * Серверная страховка переписана на REST-хук `rest_after_insert_{post_type}` с прямым доступом к объекту `WP_REST_Request` — больше никакого парсинга `php://input`.
  * Хук `save_post` теперь обрабатывает только классический редактор (REST-контекст исключён).
  * В `rest_after_insert_*` используется `delete_post_meta` + `add_post_meta`, чтобы исключить эффект «удаления значения, совпадающего с дефолтом» в WordPress.
  * В `assets/gutenberg-panel.js` тумблер пишет/читает новый ключ `wp_ru_max_skip` со старым ключом как fallback.
  * В `WP_Ru_Max_Post_Sender` поддержка обоих ключей с приоритетом нового.
  * Версия Gutenberg-скрипта теперь = `filemtime(gutenberg-panel.js)` — браузер автоматически подхватывает свежий JS даже без bump'а версии плагина.
* Изменено поведение по умолчанию: автоотправка в MAX теперь ВЫКЛЮЧЕНА для всех новых статей. Раньше тумблер по умолчанию стоял в положении ВКЛ — это приводило к случайной отправке черновиков. Теперь автор должен явно ВКЛЮЧИТЬ автоотправку для нужной статьи перед публикацией. Дефолт `register_post_meta` для `wp_ru_max_skip` изменён с `'0'` на `'1'`. Логика `WP_Ru_Max_Post_Sender::on_post_status_change` теперь отправляет сообщение только если значение мета явно `'0'`.


### 1.0.23
- **Исправлено (критично):** переключатель **«Автоотправка в MAX»** в редакторе Гутенберг не сохранял положение «ВЫКЛ» — после нажатия «Сохранить» и обновления страницы тумблер самопроизвольно возвращался в положение «ВКЛ». Теперь состояние сохраняется и восстанавливается корректно.

  - Значения нормализованы: всегда сохраняется строка `'1'` (выключить автоотправку) или `'0'` (включить автоотправку) — раньше использовалась пустая строка, которая совпадала со значением по умолчанию и могла удаляться WordPress'ом.
  - Логика чтения в `WP_Ru_Max_Post_Sender` обновлена для обратной совместимости со старыми значениями.

## FAQ

**Где взять токен бота MAX?**  
[max.ru/partner](https://max.ru/partner) → «Чат-боты» → «Интеграция» → «Получить токен».

**Как узнать ID канала или группы?**  
Публичный канал — никнейм с `@` (например, `@my_channel`).  
Группа — числовой ID (получить через бота `@get_id_bot` в MAX).

**Работает ли плагин без лицензии?**  
Базовые функции доступны. Полный доступ требует лицензионного ключа.

**Почему звук не играет при загрузке страницы?**  
Браузеры блокируют автовоспроизведение звука до первого взаимодействия пользователя. Звук сыграет сразу после первого клика, касания или прокрутки.

**Работает ли обновление без WordPress.org?**  
Да. Плагин проверяет GitHub Releases и обновляется стандартными средствами WordPress.
