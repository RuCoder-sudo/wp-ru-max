# Changelog for WP Ru-max

Все значимые изменения в этом проекте будут задокументированы в этом файле.

Формат основан на [Keep a Changelog](https://keepachangelog.com/ru/1.0.0/),
и этот проект придерживается [Semantic Versioning](https://semver.org/lang/ru/).

## [1.0.40] - 2026-06-30

### Changed
- **Адрес MAX API** — константа `WP_RU_MAX_API_BASE` изменена с `https://platform-api.max.ru` на `https://platform-api2.max.ru` согласно официальному уведомлению MAX для бизнеса (обязательный переход до 19 июля 2026).
- **Поддержка сертификата Минцифры** — в основном методе `request()` класса `WP_Ru_Max_API` добавлен параметр `'sslverify' => false`. Новый адрес `platform-api2.max.ru` использует сертификат Минцифры России, которого нет в стандартном CA-bundle WordPress/cURL, из-за чего без этого параметра все запросы к API падали бы с SSL-ошибкой. Метод `request_multipart()` и загрузка изображений уже имели `sslverify => false` — теперь поведение единообразно во всех методах.

## [1.0.39] - 2026-06-20

### Added
- **Хэштеги и упоминания** — новый блок «Хэштеги и упоминания» в настройках «Отправка публикаций». Теги WordPress автоматически преобразуются в хэштеги MAX (`#тег`) и добавляются в конец сообщения. Поддерживается поле «Упоминания менеджеров» — список `@username` через запятую, которые будут добавлены в каждое сообщение о публикации.
- **Интерфейс** — вкладки приборной панели перенесены в левую боковую панель WordPress (подменю) для удобной навигации.

### Fixed
- **chat_id формат** — метод `normalize_chat_id()` в классе API: строка формата `id{user_id}_{bot_id}_bot` корректно нормализуется в числовой ID пользователя. Устраняет ошибки отправки при нестандартном формате chat_id.

## [1.0.38] - 2026-06-20

### Added
- **UTM-метки для чат-виджета** — новый блок «UTM-метки» в настройках чат-виджета. Параметры `utm_source`, `utm_medium`, `utm_campaign`, `utm_content` автоматически добавляются к ссылке виджета.
- **Яндекс.Метрика для чат-виджета** — новый блок «Яндекс.Метрика». При клике на иконку виджета вызывается `ym(counter, 'reachGoal', goal)`. Настройки: номер счётчика и идентификатор цели.
- **Общая защита от дублей уведомлений** — новый блок «Общая защита от дублей» в разделе «Личные уведомления». Решает проблему двойных Telegram-уведомлений для MPHB, Bookly, YClients: второй email с тем же `#ID` в теме пропускается в течение заданного окна (по умолчанию 30 сек).

### Fixed
- **Ссылки на платформу MAX** — все вхождения `https://max.ru/partner` заменены на `https://business.max.ru` в соответствии с текущим URL личного кабинета MAX для партнёров.

## [1.0.37] - 2026-06-08

### Added
- **Вертикальный отступ виджета** — слайдер «Высота значка над низом экрана» увеличен до 300 px с подсказкой для случаев, когда нижнее меню темы перекрывает значок чата.

## [1.0.36] - 2026-06-05

### Added
- **Поддержка bbPress форумов** — плагин корректно обрабатывает топики bbPress (тип записи `topic`), у которых нет панели миниатюры.
- **Автоматический поиск изображений** — новый метод `get_post_image_url()`: 1) миниатюра записи → 2) первый `<img>` из тела контента → 3) прикреплённый медиафайл. Если ни одного нет — отправка идёт без изображения без ошибок.

## [1.0.35] - 2026-06-05

### Fixed
- **Критично: загрузка изображений** — метод запроса к MAX Upload API исправлен с `GET` на `POST /uploads?type=image` (в соответствии с актуальной документацией MAX API). Ошибка `Path /uploads is not recognized` устранена.
- **Резервный метод загрузки** — добавлен прямой multipart-POST файла на `/uploads?type=image` как основной метод (Метод A). Двухшаговая загрузка через pre-signed URL оставлена как Метод B.
- **Три уровня отказоустойчивости** при отправке изображений: Метод A (прямой multipart) → Метод B (двухшаговый POST) → URL → только текст.
- Исправлена ошибка `Failed to upload image` в запасном методе по URL — добавлено детальное логирование причины отказа.

### Added
- **Подробное логирование загрузки изображений** — в журнале теперь отображаются все этапы: скачивание, метаданные файла, попытки загрузки, точный ответ MAX API (HTTP-код, тело ответа).
- **Логирование ответов API** — каждый запрос к MAX API теперь записывает `response_raw` и HTTP-код в лог при ошибках.
- Новый приватный метод `request_multipart()` — унифицированная загрузка файлов через multipart/form-data с полным логированием.

### Changed
- Версия плагина обновлена до 1.0.35.

## [1.0.34] - 2026-05-30

## [1.0.32] - 2026-05-25

### Added
- **Поддержка WordPress Multisite** — плагин теперь корректно работает в сетях WordPress (мультисайт, поддомены, поддиректории).
- **Сетевая активация** — при активации плагина для всей сети (Сеть → Плагины → Активировать сетево) автоматически создаются таблицы базы данных и настройки по умолчанию для каждого подсайта.
- **Сетевая лицензия** — новая страница «Сеть → Ru-max (Сеть)» в панели суперадминистратора. Одна лицензия покрывает все подсайты сети без необходимости активировать плагин на каждом сайте отдельно.
- **Поддержка поддоменов** — лицензия на корневой домен (`example.com`) автоматически распространяется на все поддомены (`shop.example.com`, `blog.example.com` и т.д.).
- **Таблица подсайтов** — на странице сетевых настроек отображается статус лицензии и наличие токена бота для каждого подсайта сети.
- **Хук для новых подсайтов** — при добавлении нового подсайта в сеть таблица истории создаётся автоматически.
- **Особые условия для НКО и социальных проектов** — бессрочная бесплатная лицензия для организаций сферы социального обслуживания, благотворительности, образования, здравоохранения, культуры и спорта (ФЗ № 7-ФЗ, № 135-ФЗ). По письменному запросу с подтверждением статуса НКО/ОКВЭД. Ограничение: один домен на организацию.


### Changed
- Версия плагина обновлена до 1.0.32.
- Заголовок плагина содержит `Network: true` для поддержки сетевой активации.
- Метод `activate()` принимает параметр `$network_wide` — при сетевой активации создаёт таблицы на всех существующих подсайтах.
- Метод `uninstall()` корректно удаляет данные со всех подсайтов при удалении плагина.
- Описание плагина обновлено: добавлено упоминание поддержки Multisite.
- Папка assets реорганизована: CSS-файлы → `assets/css/`, JS-файлы → `assets/js/`. Все пути в PHP обновлены.
- Совместимость: WordPress 5.8 – 6.7, PHP 7.4 – 8.3.

### Fixed
- **Безопасность лицензии**: режим `scope=subdomain` строго ограничен поддоменами того же домена. Лицензия `example.com` покрывает `shop.example.com` ✓ и не покрывает `evil.com` ✗.
- **Сетевая лицензия**: `scope=network` (через «Сеть → Ru-max (Сеть)») корректно покрывает все подсайты сети без ограничения по домену.

## [1.0.31] - 2026-05-22

### Added
- **Кнопка «Поделиться в MAX»** (Дополнительные настройки) — тумблер включения/выключения кнопки «Поделиться в MAX» в конце каждой статьи и страницы. Ссылка формируется автоматически с адресом и заголовком материала (`max.ru/share?url=…`). Кнопка стилизована в фирменном синем цвете MAX.
- **MAX-авторизация (OAuth)** (Дополнительные настройки) — тумблер включения входа на сайт через аккаунт мессенджера MAX. При включении кнопка «Войти через MAX» появляется на странице входа WordPress, странице регистрации, в форме входа WooCommerce и на странице оформления заказа WooCommerce. Пользователи автоматически привязываются по MAX ID или email; при первом входе создаётся аккаунт WordPress с ролью «Подписчик» и автоматически заполняется имя и аватар.
- В разделе MAX-авторизации добавлены поля **App ID** и **App Secret** (получаются на max.ru/partner) и отображается Redirect URI для настройки OAuth-приложения на стороне MAX.

### Changed
- Версия плагина обновлена до 1.0.31 во всех файлах.

## [1.0.29] - 2026-04-15

### Added
- Раздел «Отправка push-уведомлений в MAX» во вкладке «Отправка публикаций» — отправка произвольного текста прямо в канал/группу MAX одним кликом без создания записи.

- Rate-limiting для push-отправок — не более 10 отправок в минуту на пользователя (защита от случайного спама).
- Кэширование ответа API `/me` через WordPress transient на 5 минут — уменьшает количество внешних запросов при частой проверке подключения.
